The decision came down to query patterns. With AsyncStorage, every read loads the entire value for a key. For user preferences (theme, font size, last read position), that's fine. For content with relationships, it's a bottleneck.

SQLite gave us:
- Indexed queries across 31K+ verses
- JOINs for cross-references without loading everything into memory
- Transactions for batch operations (importing reading plans)
- WAL mode for concurrent reads during sync

The tradeoff: more complex setup, raw SQL strings instead of a nice API, and we had to handle migrations manually. But the performance difference was not close. Loading a chapter went from ~400ms to ~8ms.
